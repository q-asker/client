# /pr — PR 작성 스킬

사용자와 대화하며 함께 PR을 완성하는 스킬. AI가 전적으로 작성하는 것이 아니라, 변경사항을 하나하나 짚어가며 사용자가 의도와 맥락을 설명하도록 유도한다.

## 트리거

`/pr` 명시적 호출만

## 워크플로우

### 0단계: 사전 분석

아래 작업을 **병렬로** 실행하여 정보를 수집한다:

**Git 정보 수집 (Bash):**

```bash
git branch --show-current
git log develop..HEAD --oneline
git diff develop...HEAD --stat
git diff develop...HEAD
```

**기존 PR 감지 (GitHub MCP):**

`mcp__github__list_pull_requests`로 현재 브랜치의 열린 PR을 조회한다:

- `owner`: 리포지토리 소유자
- `repo`: 리포지토리 이름
- `head`: `{owner}:{현재브랜치명}`
- `state`: `open`

> owner/repo는 `git remote get-url origin`에서 추출한다.

수집한 정보를 바탕으로 변경 내용을 파악한다.

**기존 PR이 존재하는 경우:**

- 사용자에게 알린다: `"이 브랜치에 이미 PR #N이 있습니다. 기존 내용을 참고해서 재작성할게요."`
- `mcp__github__get_pull_request`로 기존 PR의 상세 본문(body)을 가져와 참고 자료로 활용한다 — 1~2단계 대화 시 기존 설명을 기반으로 질문한다
- 5단계에서 `mcp__github__create_pull_request` 대신 `mcp__github__update_issue`를 사용한다
- **미디어 자산 보존:** 기존 PR 본문에서 링크(`[text](url)`)와 이미지(`![alt](url)`, `<img>` 태그)를 모두 추출하여 별도로 보관한다

### 1단계: 변경사항별 대화 (핵심)

커밋과 diff를 **논리적 변경 단위**로 그룹핑한다. 각 그룹에 대해 순서대로 사용자에게 질문한다:

```
"이 변경은 [요약]인데, 왜 이렇게 했는지 설명해주세요"
```

- 사용자가 답변하면 → 핵심을 정리하고 → 다음 변경사항으로 넘어간다
- trade-off가 있는 변경의 경우 추가 질문:
  ```
  "여기서 다른 선택지도 있었을 텐데, 왜 이 방식을 골랐는지?"
  ```
- 모든 변경사항을 다 다룰 때까지 반복한다
- **AI 인사이트 제공:** 각 변경사항에 대해 사용자의 설명을 듣고 정리한 뒤, AI가 diff 분석을 통해 발견한 추가 관점(설계 의도, 잠재적 리스크, 개선 포인트 등)을 함께 제시한다. 사용자는 이를 검토하여 PR 본문에 반영할지 결정한다.

**주의:** 한 번에 모든 질문을 던지지 않는다. 한 변경사항씩 순서대로 대화한다.

### 2단계: 체크리스트 대화

1단계에서 다룬 각 변경 단위에 대해, 리뷰어가 직접 재현하며 확인할 수 있는 방법을 사용자에게 질문한다:

```
"이 변경을 리뷰어가 어떻게 확인하면 될까요? (재현 경로, 확인 포인트 등)"
```

- 사용자가 답변하면 → 체크리스트 항목으로 정리하고 → 다음 변경사항으로 넘어간다
- 재현이 불필요한 변경(삭제, 설정 변경 등)은 사용자에게 확인 후 생략할 수 있다
- 모든 변경사항을 다 다룰 때까지 반복한다

**주의:** 1단계와 동일하게, 한 변경사항씩 순서대로 대화한다.

### 3단계: PR 본문 초안 생성

대화 내용을 바탕으로 `.github/PULL_REQUEST_TEMPLATE.md` 템플릿에 맞게 초안을 작성한다.

- 각 논리적 변경 단위를 `### 변경 N: [제목]` 형식으로 작성
- trade-off가 있는 변경만 `#### 의사 선택과정 (trade-off)` 섹션 포함
- trade-off가 없는 변경은 해당 섹션 생략
- 체크리스트에 2단계에서 수집한 재현 확인 항목을 포함한다
- **미디어 자산 재배치 (기존 PR 재작성 시):**
  - 0단계에서 보존한 링크/이미지를 새 본문의 관련 변경사항 섹션에 원래 위치대로 재배치한다
  - 새 본문에 대응하는 섹션이 없거나 더 이상 관련 없는 미디어 자산은 본문 하단에 `## Deprecated` 섹션을 만들어 몰아넣는다
  - `## Deprecated` 섹션은 해당 자산이 있을 때만 생성한다
- 마크다운으로 미리보기를 제시한다

### 4단계: 피드백 & 수정

- 사용자에게 초안을 보여주고 피드백을 요청한다
- 수정 요청이 있으면 반영 → 다시 미리보기
- 사용자가 **"확정"** 또는 동의할 때까지 반복한다

### 5단계: PR 생성/수정

사용자가 확정하면:

**새 PR인 경우:**

`mcp__github__create_pull_request`를 사용한다:

- `owner`: 리포지토리 소유자
- `repo`: 리포지토리 이름
- `title`: PR 제목
- `head`: 현재 브랜치명
- `base`: `develop`
- `body`: PR 본문 내용

**기존 PR이 있는 경우:**

`mcp__github__update_issue`를 사용한다 (PR은 Issue의 superset이므로 동일 도구로 수정 가능):

- `owner`: 리포지토리 소유자
- `repo`: 리포지토리 이름
- `issue_number`: PR 번호
- `title`: PR 제목
- `body`: PR 본문 내용

- PR URL을 반환하고 완료한다
- 인라인 코멘트가 필요한 경우 사용자에게 안내한다

## PR 본문 작성 규칙

- `## 📢 설명` 아래에 변경사항을 나열
- 각 변경사항은 `### 변경 N: [제목]` 형식
- 설명은 사용자의 답변을 **기반**으로 작성하되, AI가 diff 분석에서 발견한 기술적 인사이트(설계 패턴, 잠재적 리스크, 성능 고려사항 등)도 사용자 승인 하에 반영한다
- trade-off 섹션은 선택적 — 실제 trade-off가 있는 경우만 포함
- `### 코드 설명: 인라인 코멘트 확인` 섹션 유지
- 체크리스트는 기본 항목 + 2단계에서 수집한 재현 확인 항목으로 구성한다

## 주의사항

- 사용자의 말을 그대로 옮기되, 리뷰어가 이해할 수 있도록 정리한다
- 사용자가 설명하지 못하는 변경이 있으면 "이 부분은 왜 필요했나요?"로 유도한다
- PR 제목은 브랜치명의 이슈 번호를 포함한다 (예: `[ICC-270] 기능 설명`)
