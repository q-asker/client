# /init-rules — Rules 파일 초기화

## 역할

`.claude/rules/` 디렉토리에 프로젝트 맞춤 규칙 파일을 생성하거나 보완한다.
`CLAUDE.md`가 존재하면 기술 스택 정보를 읽어 자동으로 반영한다.

---

## 실행 순서

### STEP 1 — 컨텍스트 수집

1. `CLAUDE.md` 읽기 → 기술 스택, 프레임워크 파악
2. `CLAUDE.md` 없으면 `/init` 먼저 실행 안내 후 중단
3. `.claude/rules/` 디렉토리 존재 여부 확인
   - 기존 파일 있으면 → CLAUDE.md의 기술 스택과 대조하여 불일치 여부 판별
     - **불일치** (다른 프로젝트에서 복사된 경우): 현재 프로젝트에 맞게 **전체 재작성** (기존 파일 덮어쓰기)
     - **일치** (현재 프로젝트용): 누락 항목만 **보완** (기존 내용 보존)
   - 없으면 → 전체 신규 생성
4. `/mnt/skills/` 하위 디렉토리 스캔 → 사용 가능한 스킬 목록 파악

---

### STEP 2 — 5개 파일 초안을 한 번에 생성

아래 5개 파일을 **한 번의 작업으로** 모두 생성한다.
각 파일의 내용은 STEP 1에서 수집한 프로젝트 정보에 맞게 커스터마이징한다.

#### 2-1. `task-workflow.md`

```markdown
# 작업 흐름 규칙

## 세션 시작 가드 (최우선)

**모든 작업보다 먼저** 아래 2가지를 확인한다. 실패 시 작업을 시작하지 않는다.

1. `CLAUDE.md`에 `⚠️ 초기화 미완료` 마커가 있는지 확인
   - 있으면 → 사용자에게 안내: `⚠️ 프로젝트 초기화가 완료되지 않았습니다. /init-rules를 먼저 실행하세요.`
   - 사용자가 명시적으로 "무시하고 진행해"라고 하지 않는 한 작업을 시작하지 않는다
2. `CLAUDE.md`가 없으면 → 사용자에게 안내: `⚠️ CLAUDE.md가 없습니다. /init을 먼저 실행하세요.`

## 작업 모드

| 모드         | 트리거                              | 동작                    |
| ------------ | ----------------------------------- | ----------------------- |
| 즉시 실행    | 요청이 명확하고 범위가 작을 때      | 바로 구현               |
| 계획 후 실행 | 범위가 크거나 아키텍처 변경 포함 시 | 계획 공유 → 승인 → 구현 |
| 질문 우선    | 요구사항이 모호할 때                | 핵심 질문 1~2개 → 구현  |

## 모호한 상황 대처

- UI 컴포넌트가 필요하면 **{프로젝트 UI 라이브러리}** 기준으로 구현한다.
  - TODO: 실제 UI 라이브러리로 교체 (예: shadcn/ui, MUI, Tailwind)
- 파일 경로가 불명확하면 기존 컨벤션을 따른다.
- 테스트 방식이 불명확하면 기존 테스트 파일 패턴을 따른다.

## 질문 원칙

- 한 번에 최대 2개 질문
- 구현에 직접 영향을 주는 것만 질문
- 추론 가능한 것은 질문하지 않고 결정 후 명시
```

#### 2-2. `coding-conventions.md`

기술 스택에 따라 아래 분기로 작성한다.

**TypeScript / JavaScript 계열:**

```markdown
# 코딩 컨벤션

## 네이밍

- 컴포넌트: PascalCase
- 함수/변수: camelCase
- 상수: SCREAMING_SNAKE_CASE
- 파일: kebab-case (컴포넌트는 PascalCase)

## 컴포넌트

- 함수형 컴포넌트만 사용
- Props 타입은 interface로 정의, 파일 상단에 배치
- default export 사용

## 스타일링

- TODO: 프로젝트 스타일링 방식 명시 (Tailwind / CSS Modules / styled-components)

## Import 순서

1. 외부 라이브러리
2. 내부 모듈 (절대경로)
3. 상대경로
4. 타입 import (type 키워드 사용)

## 기타

- `any` 타입 사용 금지
- 비동기는 async/await 사용 (Promise chain 금지)
```

**Python 계열:**

```markdown
# 코딩 컨벤션

## 네이밍

- 클래스: PascalCase
- 함수/변수: snake_case
- 상수: SCREAMING_SNAKE_CASE
- 파일: snake_case

## 타입

- 모든 함수에 타입 힌트 필수
- `Any` 타입 사용 금지

## Import 순서 (isort 기준)

1. 표준 라이브러리
2. 서드파티
3. 내부 모듈

## 기타

- Docstring: Google 스타일
- 최대 줄 길이: 88자 (black 기본값)
```

### STEP 3 — CLAUDE.md 초기화 마커 제거

`CLAUDE.md`에 아래 블록이 존재하면 **제거**한다:

```markdown
> **⚠️ 초기화 미완료 — `/init-rules`를 실행하세요.**
> 현재 rules 파일이 이 프로젝트에 맞지 않습니다. 작업 전에 반드시 `/init-rules`를 실행하세요.
> 이 블록은 `/init-rules` 완료 시 자동으로 제거됩니다.
```

이 마커는 `/init`이 rules 불일치/미존재 시 삽입한 것이다.
제거 후 빈 줄이 연속되면 하나로 정리한다.

### 상태 메시지 결정 규칙

**3단계 `/plan-feature` 상태:**

- `docs/PRD.md` 없음 → `⚠️ PRD 없음 — 반드시 실행`
- `docs/PRD.md` 있고 `docs/roadmaps/` 없음 → `⚠️ ROADMAP 없음`
- 둘 다 존재 → `✅ PRD+ROADMAP 존재 — 건너뛰기 가능`

### 출력 규칙

- **프로그레스 바**는 리포트 최상단에 배치한다
- **"세팅 → 규칙 → 계획"** 니모닉은 프로그레스 바 바로 아래에 배치한다
- **리포트의 마지막 커맨드 안내는 반드시 `> 다음: /plan-feature`**로 출력한다 (PRD+ROADMAP 모두 있으면 `> 초기화 완료! 이제 개발을 시작하세요.`)
