# /pr — PR 작성 스킬

사용자와 대화하며 함께 PR을 완성하는 스킬. AI가 전적으로 작성하는 것이 아니라, 변경사항을 하나하나 짚어가며 사용자가 의도와 맥락을 설명하도록 유도한다.

## 트리거

`/pr` 명시적 호출만

## 워크플로우

### 0단계: 사전 분석

아래 명령어를 **병렬로** 실행하여 정보를 수집한다:

```bash
git branch --show-current
git log develop..HEAD --oneline
git diff develop...HEAD --stat
git diff develop...HEAD
```

수집한 정보를 바탕으로 변경 내용을 파악한다.

### 1단계: 변경사항별 대화 (핵심)

커밋과 diff를 **논리적 변경 단위**로 그룹핑한다. 각 그룹에 대해 순서대로 사용자에게 질문한다:

```
"이 변경은 [요약]인데, 왜 이렇게 했는지 설명해주세요"
```

- 사용자가 답변하면 → 핵심을 정리하고 → 다음 변경사항으로 넘어간다
- trade-off가 있는 변경의 경우 추가 질문:
  ```
  "여기서 다른 선택지도 있었을 텐데, 왜 이 방식을 골랐는지?"
  ```
- 모든 변경사항을 다 다룰 때까지 반복한다

**주의:** 한 번에 모든 질문을 던지지 않는다. 한 변경사항씩 순서대로 대화한다.

### 2단계: PR 본문 초안 생성

대화 내용을 바탕으로 `.github/PULL_REQUEST_TEMPLATE.md` 템플릿에 맞게 초안을 작성한다.

- 각 논리적 변경 단위를 `### 변경 N: [제목]` 형식으로 작성
- trade-off가 있는 변경만 `#### 의사 선택과정 (trade-off)` 섹션 포함
- trade-off가 없는 변경은 해당 섹션 생략
- 마크다운으로 미리보기를 제시한다

### 3단계: 피드백 & 수정

- 사용자에게 초안을 보여주고 피드백을 요청한다
- 수정 요청이 있으면 반영 → 다시 미리보기
- 사용자가 **"확정"** 또는 동의할 때까지 반복한다

### 4단계: PR 생성

사용자가 확정하면:

```bash
gh pr create --base develop --title "PR 제목" --body "$(cat <<'EOF'
PR 본문 내용
EOF
)"
```

- PR URL을 반환하고 완료한다
- 인라인 코멘트가 필요한 경우 사용자에게 안내한다

## PR 본문 작성 규칙

- `## 📢 설명` 아래에 변경사항을 나열
- 각 변경사항은 `### 변경 N: [제목]` 형식
- 설명은 사용자의 답변을 기반으로 작성 (AI가 임의로 꾸미지 않음)
- trade-off 섹션은 선택적 — 실제 trade-off가 있는 경우만 포함
- `### 코드 설명: 인라인 코멘트 확인` 섹션 유지
- 체크리스트는 템플릿 기본값 사용

## 주의사항

- 사용자의 말을 그대로 옮기되, 리뷰어가 이해할 수 있도록 정리한다
- 사용자가 설명하지 못하는 변경이 있으면 "이 부분은 왜 필요했나요?"로 유도한다
- PR 제목은 브랜치명의 이슈 번호를 포함한다 (예: `[ICC-270] 기능 설명`)
