# 문서 연동 규칙

코드 변경 시 **문서 분류 체계**와 **책임 영역 패턴**에 따라 관련 문서를 갱신합니다. 모든 정보는 **단 하나의 진실의 원천**에만 존재해야 합니다.

---

## 1️⃣ 문서 분류 체계

### 문서 타입 3가지

#### **Type A: 진실의 원천 (Source of Truth)**

- 정보의 **최종 권위자** — 다른 모든 문서가 참조/상속하는 기준
- 가장 **최신**, **상세**, **완전한** 정보만 포함
- 예: `CLAUDE.md` (기술 스택), `docs/PRD.md` (기능 명세), `docs/roadmaps/` (현재 상태)

**자동 로드 (매 세션 보장):**
| 문서 | 담당 영역 |
|-----|--------|
| `CLAUDE.md` | 기술 스택, 명령어, 아키텍처, 디렉토리 구조, 환경 변수, 개발 도구 |
| `code-rules.md` | 코딩 컨벤션, FSD 규칙, 네이밍, 제약 사항, import 규칙 |
| `workflow.md` | 세션 가드, 작업 모드, 에이전트/스킬 위임, 작업 원칙 |

**필요 시 읽기 (명시적 Read 필요):**
| 문서 | 담당 영역 |
|-----|--------|
| `docs/PRD.md` | 기능 명세, 데이터 모델, 비즈니스 요구사항 |
| `docs/roadmaps/ROADMAP_v*.md` | Phase/Task 계획, 진행 상태, 개발 워크플로우 |

---

#### **Type B: 계층 문서 (Hierarchical Documents)**

- **상위 진실의 원천을 상속** 후, **추가 제약/세부사항**만 작성
- 상위 문서의 내용을 **반복하지 않음** — 참조로 명시만 함
- 예: `design-variants-rules.md` → `shrimp-rules.md` 상속, `solve-quiz-design-rules.md` → `design-variants-rules.md` 상속

**특징:**

- 문서 상단에서 상위 문서 명시: `shrimp-rules.md의 Import 규칙을 따릅니다`
- 상위 규칙은 반복하지 말고 참조만 함
- 새로운 제약/구체화만 추가

---

#### **Type C: 절차/가이드 문서 (Process/Guide Documents)**

- **어떻게 사용할지, 어떤 순서로 할지** 설명 → 실행 절차 중심
- 진실의 원천을 **참조만 함** — 기술 규칙은 담당하지 않음
- 예: 스킬 사용 가이드 (design-variants.md), PR 템플릿

**특징:**

- 기술 규칙의 반복 금지 → "자세한 규칙은 {링크}를 참고하세요"
- 절차적 흐름에 집중 (1단계 → 2단계 → ... → 검증)

---

### 문서 타입 결정 트리

```
새 문서를 추가하려 할 때:

1. "이 문서의 정보가 다른 곳에도 있는가?"
   → YES: 기존 문서에 추가 (Type B 계층 문서 고려)
   → NO: 새 문서 필요

2. "이 문서가 다른 문서를 기반으로 하는가?"
   → YES: Type B (계층 문서) — 상위 문서 명시하고 추가 내용만 작성
   → NO: 진행 ↓

3. "이 문서가 절차나 사용 방법을 설명하는가?"
   → YES: Type C (절차 가이드) — 기술 규칙은 다른 문서 참조
   → NO: Type A (진실의 원천)
```

## 2️⃣ 책임 영역 패턴 (Responsibility Pattern)

### 변경 타입별 대응 문서 타입

구체적 파일명 대신 **문서 타입과 책임 영역**으로 매핑합니다:

| 변경 타입                         | 대응 문서 타입       | 대응 문서 예시                                           | 원칙                             |
| --------------------------------- | -------------------- | -------------------------------------------------------- | -------------------------------- |
| **기술 스택 / 의존성 변경**       | Type A (아키텍처)    | `CLAUDE.md`                                              | 프로젝트 전체 영향도 정의        |
| **아키텍처 / 디렉토리 구조 변경** | Type A (아키텍처)    | `CLAUDE.md`                                              | 상위 개념 정의                   |
| **일반 기술 규칙 / 제약 추가**    | Type A (기술 규칙)   | `code-rules.md`, `rules/*.md`                            | 모든 개발자에게 적용되는 규칙    |
| **특화 규칙 / 추가 제약 추가**    | Type B (계층 문서)   | `design-variants-rules.md`, `solve-quiz-design-rules.md` | 상위 규칙 명시, 추가 내용만 작성 |
| **절차 / 사용 가이드 추가**       | Type C (절차 가이드) | 스킬 문서, 가이드                                        | 기술 규칙 참조만, 반복 금지      |
| **상태 / 진행 상황 변경**         | Type A (상태)        | `ROADMAP_v*.md`, Task 시스템                             | 매 작업마다 즉시 반영            |
| **Git 커밋**                      | —                    | `/commit` 스킬                                           | 문서 동기화 체크 포함            |

---

### 중복 판단 알고리즘

같은 내용이 여러 문서에 존재하면:

1. **원인 파악**
   - Type B 계층 문서가 상위 문서 내용을 반복? → 제거하고 참조로 변경
   - Type C 절차 가이드가 기술 규칙을 설명? → 규칙은 Type A로 이동, 가이드는 참조로 변경
   - Type A 문서끼리 중복? → 어느 것이 권위인지 명확히 하고 통합

2. **통합 방향**
   - 상위 계층 문서 ← 하위 계층 문서 (하위가 상위로 이동)
   - Type A ← Type B, C (기술 규칙은 Type A만)
   - 더 구체적 범위 ← 더 일반적 범위 (구체화는 상위 규칙과 별도로)

3. **확인**
   - 통합 후 검색으로 같은 표현 재확인
   - 링크/참조 업데이트

## 3️⃣ 새 문서 추가 체크리스트

새 문서를 작성하기 전에 **반드시** 아래 3가지를 결정하세요:

### ✓ 문서 타입 결정

```
□ Type A (진실의 원천)
  → 이 정보의 최종 권위인가? 다른 곳에 더 적절한 위치가 없는가?

□ Type B (계층 문서)
  → 상위 문서가 있는가? 그곳을 명시할 것인가?
  → 상위 규칙은 제외하고 추가 제약만 포함시켰는가?

□ Type C (절차 가이드)
  → 기술 규칙은 포함하지 않았는가? (다른 Type A 문서 참조로만 언급)
```

### ✓ 계층 관계 명시 (Type A 또는 Type B인 경우)

```
□ 의존하는 상위 문서 기록:
  - 상위 진실의 원천: ___________
  - 상속/참조 관계: ___________

□ 이 문서가 구체화하는 상위 개념: ___________

□ 상위 문서 내용 반복 확인:
  - [ ] 기술 규칙 반복 없음
  - [ ] import 규칙 반복 없음
  - [ ] 네이밍 규칙 반복 없음
  - [ ] 기타 ___________ 반복 없음
```

### ✓ 책임 영역 명시

```
이 문서가 담당:
- [ ] ...
- [ ] ...

이 문서가 담당하지 않음 (다른 문서에서 처리):
- [ ] ...
- [ ] ...
```

## 4️⃣ 계층 구조 유지 규칙

### 상속 패턴

**좋은 예:**

```
shrimp-rules.md (Type A)
└─ FSD 규칙, Import 규칙, 포맷팅, 네이밍, 색상 제약 (일반)

  design-variants-rules.md (Type B)
  └─ 상위 규칙을 따릅니다 (생략하고 참조만)
  └─ 추가 제약: 파일 크기, 애니메이션 라이브러리, 반응형, 다크모드 (변형 특화)

    solve-quiz-design-rules.md (Type B)
    └─ design-variants-rules.md를 상속
    └─ 추가 내용: 변형별 레이아웃, 색상 팔레트, 애니메이션 패턴 (solve-quiz 특화)

  design-variants.md (Type C)
  └─ 절차: 5단계 스킬 사용법
  └─ 참조: design-variants-rules.md, solve-quiz-design-rules.md (기술 규칙은 여기서)
```

**나쁜 예:**

```
❌ design-variants.md에 색상 규칙, import 규칙 반복
❌ design-variants-rules.md에 shrimp-rules.md 내용 복사
❌ 같은 기술 규칙이 여러 문서에 다른 표현으로 존재
```

## 5️⃣ 준수 원칙

### 코드 변경 시

- 코드 변경과 **문서 갱신은 동일 작업 단위**로 수행
- 구체적 파일명 대신 **문서 타입과 책임 영역**으로 판단
- 기술 스택/아키텍처 변경 시 `CLAUDE.md` 먼저 갱신

### 문서 작성 시

- **새 문서 추가 체크리스트** 완료 후 작성 시작
- Type B 문서는 **상위 문서 명시** 필수 (문서 상단)
- Type C 문서는 **기술 규칙 반복 금지** — 참조로만 언급
- 중복 발견 시 **상위 문서로 통합** (하위는 참조로 변경)

### 의심스러울 때

1. 진실의 원천이 어디인지 확인: 가장 추상적 수준인 `CLAUDE.md`, `code-rules.md`부터 체크
2. 계층 관계 시각화: 이 문서가 의존하는 상위 문서가 있는가?
3. 중복 검색: 같은 내용이 다른 문서에도 있는가?

## 참고: 문서 확장 예시

**향후 새 페이지 변형 추가 시:**

1. `{page}-design-rules.md` 생성 (Type B, design-variants-rules.md 상속)
2. `design-variants-rules.md` 업데이트 불필요 (일반 규칙이므로)
3. design-variants.md의 "변형 체계" 섹션만 업데이트 (참조 추가)

**새 기술 규칙 추가 시:**

1. `CLAUDE.md` 또는 `code-rules.md`에 추가할지 결정
2. 이미 존재하는 계층 문서들에서 중복 검색
3. 반복되는 내용 제거, 참조로 변경
