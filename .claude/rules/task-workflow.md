# 작업 흐름 규칙

## 세션 시작 가드 (최우선)

**모든 작업보다 먼저** 아래 2가지를 확인한다. 실패 시 작업을 시작하지 않는다.

1. `CLAUDE.md`에 `⚠️ 초기화 미완료` 마커가 있는지 확인
   - 있으면 → 사용자에게 안내: `⚠️ 프로젝트 초기화가 완료되지 않았습니다. /init-rules를 먼저 실행하세요.`
   - 사용자가 명시적으로 "무시하고 진행해"라고 하지 않는 한 작업을 시작하지 않는다
2. `CLAUDE.md`가 없으면 → 사용자에게 안내: `⚠️ CLAUDE.md가 없습니다. /init을 먼저 실행하세요.`

## 로드맵 기반 작업 흐름

- **모든 작업은 `docs/roadmaps/`의 최신 ROADMAP을 기준으로 계획하고 실행한다** (`[COMP]` 접두사가 없는 파일이 현재 활성 로드맵)
- 작업 시작 전 반드시 활성 로드맵을 읽고 현재 Phase와 Task 진행 상태를 파악한다
- Phase 순서를 준수하며, 각 Phase 내 Task도 명시된 순서를 따른다
- 이전 Phase의 모든 Task가 완료되어야 다음 Phase로 진행할 수 있다
- 작업 완료 시 활성 로드맵에서 해당 Task를 완료 표시로 변경한다
- 각 단계 완료 후 중단하고 추가 지시를 기다린다

## 작업 우선순위

1. 활성 로드맵의 Phase/Task 순서를 최우선으로 따른다
2. 현재 Phase 내에서 "우선순위" 표시가 있는 Task를 먼저 처리한다
3. 의존성이 있는 Task는 선행 Task 완료 후 진행한다

## 작업 모드

| 모드         | 트리거                              | 동작                    |
| ------------ | ----------------------------------- | ----------------------- |
| 즉시 실행    | 요청이 명확하고 범위가 작을 때      | 바로 구현               |
| 계획 후 실행 | 범위가 크거나 아키텍처 변경 포함 시 | 계획 공유 → 승인 → 구현 |
| 질문 우선    | 요구사항이 모호할 때                | 핵심 질문 1~2개 → 구현  |

- **Plan 모드** (가벼운 작업): 파일 1~2개 수정, 단일 기능 변경, 버그 수정, 리팩토링
- **`/plan-feature` 스킬** (복잡한 작업): 새 기능 추가, 3개 이상 파일 변경 예상, PRD/ROADMAP 갱신 필요, 아키텍처 변경 수반
- Plan 모드 진행 중 작업이 예상보다 커지면 중단하고 `/plan-feature` 전환을 제안한다

## 적극적 질문 원칙

작업을 올바르게 수행하기 위해 **추측보다 질문을 우선**한다.

- **요구사항이 모호하면 즉시 질문한다** — 잘못된 방향으로 구현한 뒤 되돌리는 비용이 질문 한 번보다 크다
- **여러 구현 방식이 가능할 때 선택지를 제시하고 사용자에게 결정을 요청한다** — 임의로 선택하지 않는다
- **기존 코드와 충돌하거나 아키텍처 변경이 필요한 경우 먼저 확인한다**
- **작업 범위가 불명확하면 구체적으로 범위를 확인한다**
- 질문할 때는 간결하고 구체적으로 — 불필요하게 긴 설명 없이 핵심만 묻는다

## 모호한 상황 대처

- 활성 로드맵에 명시되지 않은 작업은 사용자에게 확인 후 진행한다
- 기술적 선택이 필요한 경우 `CLAUDE.md`의 기술 스택을 우선 따른다
- UI 디자인 결정은 shadcn/ui 기본 스타일을 따른다
- 파일 경로가 불명확하면 FSD 구조와 기존 컨벤션을 따른다
